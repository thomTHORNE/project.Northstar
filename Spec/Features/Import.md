# Import

## What it is

Import is how existing music gets into Northstar without starting from scratch. It covers two paths: connecting a streaming service account to pull in a full library, and pasting a URL to add a single item from any supported source.

Either way, once something is imported it belongs to Northstar's library. The source is recorded as a `source_link` — a reference Northstar uses to play the audio. Everything else: the structure, the relationships, the metadata — is yours.

---

## Paths

### Service import

The user connects a streaming service account and Northstar pulls in their existing library: artists, albums, tracks, and playlists, with the relationships between them preserved.

The imported data is a starting point. The user owns it from the moment it lands — they can rename things, restructure them, add notes and tags, or delete anything they don't want to keep.

**What gets imported:**

| Entity | What is pulled |
|---|---|
| Tracks | Title, duration, artist reference, album reference, source link |
| Artists | Name, source link |
| Albums | Title, artist reference, release date, cover art, source link |
| Playlists | Name, track list with order preserved |

- All imported entities are created with `pending_review: false`. Service import is a deliberate user action — nothing is flagged for review.
- Relationships between entities are preserved: albums reference their artist, tracks reference their artist and album.
- Playlists are imported as manual playlists. Tag filters are not inferred or generated — those are always user-defined.
- If a track appears in multiple albums (e.g., a studio album and a greatest hits compilation), both albums are imported and the track references both via `album_ids`.

**Duplicate handling:**

If the user runs a service import more than once, or imports from a second service that shares some of the same music, Northstar checks for existing entries before creating new ones.

- Matching is done by source link. If an entity with the same source link already exists in the library, it is not duplicated — the existing entry is used.
- If the same track exists under a different source link (e.g., the same song on Spotify and Apple Music), it is treated as a new entry. The user is responsible for merging if they choose to.

**What is not imported:**

- Streaming service ratings, recommendations, or algorithmic data
- Listening history from the service (ListeningEvents are only recorded by Northstar going forward)
- Tags or notes — these are always user-created

---

### Link import

The user pastes a URL to a track, album, artist, or playlist on any supported source. Northstar extracts what metadata it can from the link and creates the corresponding entry in the library.

Supported link types:
- Track, album, or artist on a supported streaming service
- YouTube video (treated as a track)
- A file or folder in a connected cloud storage account

**What happens after a link is pasted:**

1. Northstar identifies the link type and source.
2. Metadata is extracted: title, artist, album, duration, cover art — whatever the source makes available.
3. The entity is created in the library with a `source_link` pointing back to the original URL.
4. If the entity type implies relationships (e.g., a track has an artist and album), Northstar attempts to create or reuse those too.

**Relationship handling on link import:**

- If a linked track references an artist that already exists in the library (matched by name), the existing artist is reused. No duplicate is created.
- If the artist does not exist, a new artist entry is created.
- The same logic applies to albums.
- All newly created entities are created with `pending_review: false`. Link import is a deliberate action.

**Partial metadata:**

Not all sources provide complete metadata. A YouTube link may have a title but no artist or album. A cloud storage file may have only a filename.

- Entities created with incomplete metadata are flagged as needing attention: an unlinked track (no source link), an unassigned track (no artist), or an unassigned album (no artist).
- The user can fill in the missing fields manually from the entity's detail view.

---

## States

Import does not produce entities with special states. All imported entities are immediately part of the library with `pending_review: false`.

The one exception is incomplete metadata: tracks without a source link, artist, or album are surfaced in the library as needing attention — the same mechanism used anywhere else in the library for incomplete entries. This is not an import-specific state; it is the library's standard incomplete-entry handling.

---

## Constraints

- Northstar does not store audio files. Import creates library entries and source links — not local copies of audio.
- Playlist tag filters are never generated by import. Tag filters are always defined by the user.
- Listening history is not imported from external services. ListeningEvents begin at the point Northstar starts recording.
- Duplicate resolution by name (for artist/album matching on link import) is best-effort. The user is responsible for merging entries that represent the same real-world entity but were created separately.

---

## Edge cases

| Scenario | Behavior |
|---|---|
| User runs a service import twice from the same account | Entities with matching source links are not duplicated. New items added to the service since the last import are pulled in. Existing library entries are unchanged — any notes, tags, or edits the user made are preserved. |
| A playlist imported from a service shares a name with an existing Northstar playlist | The imported playlist is created with a disambiguated name (e.g., "Road Trip (Spotify)"). The user can rename it afterwards. |
| A linked YouTube video has no artist or album metadata | A track entry is created with the video title as the track title. Artist and album fields are left empty. The track is flagged as needing attention. |
| A linked track's artist already exists in the library | The existing artist is reused. No duplicate artist is created. |
| A link points to content that is no longer available | Northstar creates the entry with whatever metadata can be extracted from the URL. The source link is stored but will fail on playback. The entry is surfaced as needing attention. |
| User imports a playlist from a service, then later edits it in the service | Northstar does not sync automatically. The Northstar playlist reflects the state at the time of import. Re-importing will add new tracks but will not remove tracks the user has kept in their Northstar version. |
| A cloud storage link points to a folder | Each audio file in the folder is imported as a separate track entry. Metadata is extracted from file tags if available. |
