# Data Model

This document defines every entity in Northstar, its fields, its relationships to other entities, and the rules that govern its behavior. All feature specs reference this document. If something isn't defined here, it doesn't exist in the system.

---

## Entities

- [Track](#track)
- [Artist](#artist)
- [Album](#album)
- [Playlist](#playlist)
- [Tag](#tag)
- [Note](#note)
- [History](#history)

## Track

A track is the most granular unit of data in Northstar. It represents a single song — not the audio file itself, but everything that identifies and describes it. The actual audio lives on a streaming service; Northstar holds the reference.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID | Yes | Unique identifier |
| `title` | String | Yes | Name of the track |
| `duration` | Integer | No | Length in seconds |
| `artist_id` | UUID | No | Reference to the track's artist. Null if unassigned |
| `album_ids` | UUID[] | No | References to albums this track belongs to. Empty if not part of any album |
| `source_links` | Link[] | No | One or more links to where the track's audio can be found — a streaming service, YouTube, cloud storage, or any supported source. Empty if not yet linked |
| `pending_review` | Boolean | Yes | Whether this track was captured and is awaiting the user's accept or discard decision. Default: false |
| `created_at` | Timestamp | Yes | When the track was added to the library |
| `updated_at` | Timestamp | Yes | When the track was last modified |

### Relationships

- Belongs to one **Artist** (optional — a track without an artist is valid but flagged)
- Belongs to many **Albums** (optional — a track without any album association is valid)
- Referenced by many **Playlists**
- Has many **Tags**
- Has one **Note** (optional)

### Rules

- A track without any source links is considered **unlinked** and surfaced to the user as needing attention.
- A track without an assigned artist is considered **unassigned** and surfaced to the user as needing attention.
- A track can belong to multiple albums (e.g., a song that appears on both a studio album and a greatest hits compilation).
- A track can appear in multiple playlists.
- Deleting a track removes it from all playlists that reference it. Artists and albums are not affected.

### Deletion cascade

When a track is deleted, the user is prompted to confirm. The track is removed from all playlists that reference it. No other entities are affected. The deletion is recorded in History and can be undone.

## Artist

An artist represents the creator or performer of music. It acts as the top-level container in the library hierarchy.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID | Yes | Unique identifier |
| `name` | String | Yes | Artist or band name |
| `bio` | String | No | Optional description of the artist |
| `image_url` | String | No | URL to artist photo or image |
| `source_links` | Link[] | No | Links to the artist on any supported source — streaming service profiles, YouTube channels, or other references |
| `pending_review` | Boolean | Yes | Whether this artist was created by a capture session and is awaiting the user's accept or discard decision. Default: false |
| `created_at` | Timestamp | Yes | When the artist was added to the library |
| `updated_at` | Timestamp | Yes | When the artist was last modified |

### Relationships

- Has many **Albums**
- Has many **Tracks**
- Has one **Note** (optional)

### Rules

- An artist must have a name.
- An artist can exist in the library without any tracks (created in anticipation of adding music).
- Multiple artists with similar names are allowed — the user is responsible for resolving duplicates.

### Deletion cascade

When an artist is deleted, the user is warned and presented with a choice:

- **Delete associated tracks and albums** — removes all tracks and albums linked to this artist from the library
- **Keep tracks and albums, remove artist assignment** — tracks and albums remain in the library; tracks become unassigned, albums lose their artist reference

The chosen action and its full scope are recorded in History and can be undone as a single grouped operation.

## Album

An album is a collection of tracks released together by an artist. It preserves the original tracklist as a cohesive unit.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID | Yes | Unique identifier |
| `title` | String | Yes | Name of the album |
| `artist_id` | UUID | No | Reference to the album's artist. Null if artist was deleted and user chose to keep the album |
| `release_date` | Date | No | Official release date |
| `cover_art_url` | String | No | URL to album cover image |
| `source_links` | Link[] | No | Links to the album on any supported source — streaming services, YouTube, or other references |
| `pending_review` | Boolean | Yes | Whether this album was created by a capture session and is awaiting the user's accept or discard decision. Default: false |
| `created_at` | Timestamp | Yes | When the album was added to the library |
| `updated_at` | Timestamp | Yes | When the album was last modified |

### Relationships

- Belongs to one **Artist** (optional — an album without an artist is valid but flagged)
- Has many **Tracks** (ordered)
- Has one **Note** (optional)

### Rules

- An album must have a title.
- An album without an assigned artist is considered **unassigned** and surfaced to the user as needing attention.
- Track order within an album is preserved and reflects the original release sequence.
- An album can exist without tracks (created in anticipation of adding music).

### Deletion cascade

When an album is deleted, the user is warned and presented with a choice:

- **Delete associated tracks** — removes all tracks belonging to this album from the library
- **Keep tracks, remove album association** — tracks remain in the library as individual entries; their `album_id` is set to null

The chosen action is recorded in History and can be undone as a single grouped operation.

## Playlist

A playlist is a user-defined collection of tracks. Unlike an album, it has no fixed structure — it can be built by hand, composed dynamically by tags, or a combination of both.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID | Yes | Unique identifier |
| `name` | String | Yes | Name of the playlist. Must be unique per user |
| `description` | String | No | Optional description |
| `tag_filters` | UUID[] | No | List of tag IDs used to compose the playlist dynamically. Empty if the playlist is built entirely by hand |
| `allow_duplicates` | Boolean | Yes | Whether the same track can appear more than once. Default: false |
| `capture_mode_active` | Boolean | Yes | Whether Capture Mode is currently active on this playlist. Default: false |
| `created_at` | Timestamp | Yes | When the playlist was created |
| `updated_at` | Timestamp | Yes | When the playlist was last modified |

### Relationships

- References many **Tracks** (ordered)
- Has many **Tags** (via `tag_filters`)
- Has one **Note** (optional)

### Rules

- Playlist names must be unique per user.
- A playlist's effective contents are the union of its manually added tracks and any tracks in the library that match its tag filters. Manually added tracks appear first.
- The playlist type is inferred from its contents — it is not a field the user sets explicitly. A playlist with only manual tracks behaves as manual; one with only tag filters behaves as tag-driven; one with both behaves as mixed.
- Tag filter matches update automatically whenever tags are added or removed from tracks in the library.
- Only one playlist per user may have `capture_mode_active` set to true at any time. Enabling Capture Mode on a new playlist automatically disables it on the previously active one.
- When `allow_duplicates` is false, the same track cannot be added more than once (manually or via Capture Mode).

### Deletion cascade

When a playlist is deleted, only the playlist is removed. All tracks referenced by it remain in the library unaffected. The deletion is recorded in History and can be undone.

## Tag

A tag is a user-defined label that can be applied to any entity in the library. Tags are the primary tool for adding personal meaning and context to music, and they power tag-driven playlists.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID | Yes | Unique identifier |
| `name` | String | Yes | The label. Must be unique per user (case-insensitive) |
| `description` | String | No | Optional explanation of what the tag means |
| `color` | String | No | Optional hex color for visual identification |
| `parent_ids` | UUID[] | No | IDs of tags this tag is grouped under. Empty if top-level |
| `created_at` | Timestamp | Yes | When the tag was created |
| `updated_at` | Timestamp | Yes | When the tag was last modified |

### Relationships

- Applied to many **Tracks**, **Artists**, **Albums**, and **Playlists**
- Referenced by **Playlists** with `tag-driven` or `mixed` mode

### Rules

- Tag names must be unique per user, case-insensitive. "Chill" and "chill" are the same tag.
- A tag can be applied to any entity type — track, artist, album, or playlist.
- Renaming a tag updates all associations automatically. No confirmation required.
- Tags support a two-level hierarchy. A tag can be grouped under one or more parent tags, forming named groups (e.g., a *mood* group containing *melancholic*, *euphoric*, *focused*).
- A parent tag is just a tag — it can itself be applied to entities and used as a playlist filter.
- A tag can belong to multiple groups (many-to-many parent relationship).
- Circular references are not allowed — a tag cannot be its own ancestor.
- There is no deeper nesting beyond two levels: group → tag. This keeps the hierarchy simple and navigable.

### Deletion cascade

When a tag is deleted, the user is warned that all associations with tracks, artists, albums, and playlists will be removed, and that any tag-driven or mixed playlists using this tag will be affected. The user must confirm. The deletion and all removed associations are recorded in History and can be undone as a single grouped operation.

## Note

A note is a free-form text entry attached to a single entity in the library. It is private to the user and carries no structural meaning — it exists purely to hold personal context.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID | Yes | Unique identifier |
| `entity_type` | Enum | Yes | The type of entity this note belongs to: `track`, `artist`, `album`, `playlist`, or `tag` |
| `entity_id` | UUID | Yes | The ID of the entity this note belongs to |
| `body` | String | Yes | The content of the note. No format restrictions |
| `created_at` | Timestamp | Yes | When the note was created |
| `updated_at` | Timestamp | Yes | When the note was last modified |

### Relationships

- Belongs to exactly one entity (track, artist, album, playlist, or tag)

### Rules

- Each entity can have at most one note.
- A note is created when the user first writes in the note field for an entity, and deleted when the user clears it entirely.
- Notes are not searchable in the initial version.

### Deletion cascade

When an entity is deleted, its note is deleted with it. No separate warning is shown for the note — the entity deletion warning covers it.

## History

History is a system-level log of every change made to the user's library. It is not user-editable. It exists to give the user visibility into what has changed and the ability to undo any action.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | UUID | Yes | Unique identifier |
| `action` | Enum | Yes | The type of action: `create`, `update`, `delete` |
| `entity_type` | Enum | Yes | The type of entity affected: `track`, `artist`, `album`, `playlist`, `tag`, `note` |
| `entity_id` | UUID | Yes | The ID of the entity affected |
| `entity_snapshot` | JSON | Yes | A snapshot of the entity's state at the time of the action. Used for undo |
| `related_entries` | UUID[] | No | IDs of other History entries that are part of the same grouped operation (e.g., bulk deletion) |
| `capture_session_id` | UUID | No | If this entry was created by Capture Mode, the ID of the capture session it belongs to. Null for all other actions |
| `timestamp` | Timestamp | Yes | When the action occurred |

### Rules

- Every create, update, and delete action on any entity is logged automatically.
- Bulk operations (e.g., deleting an artist and choosing to delete all associated tracks) are logged as a group — a single undo reverses the entire operation.
- Capture Mode actions are logged with a `capture_session_id` so all entities created during a single session are identifiable and groupable in the feed.
- Capture History entries use descriptive messaging that reflects what was created: the track alone, or the track with its artist and album.
- History entries are read-only. They cannot be edited or manually deleted by the user.
- The retention window for undo eligibility is decided at the Architecture stage.
- History is presented to the user as a global chronological feed, not per-entity.
